## Context

The repo's existing tooling is described in CLAUDE.md and `.github/workflows/ci.yml`. Audit findings that motivate this change:

- `apps/web/types/api.d.ts` is regenerated by pre-push (`.husky/pre-push`) and would be regenerated again at build, but **no application file imports from it**. Every call site uses `$fetch<HandWrittenType>(...)` (see `apps/web/app/composables/useCategories.ts`, `useTags.ts`, `app/stores/auth.ts`, `app/stores/favorites.ts`, `app/pages/index.vue`, `app/pages/r/new.vue`, `app/pages/r/[slug]/index.vue`, `app/pages/r/[slug]/edit.vue`, `app/pages/favorites.vue`, `app/components/RecipeForm.vue`).
- `.github/workflows/ci.yml` web job runs `lint`, `typecheck`, `build` — but not `test`. The single Vitest spec (`apps/web/tests/useRecipeFilters.test.ts`) is therefore unenforced.
- Backend has Ruff (`apps/api/pyproject.toml`) but no static type checker — type hints exist but are never validated.
- CLAUDE.md mentions Playwright; no Playwright is installed (`apps/web/package.json` has no `@playwright/test`, no `playwright.config.ts`, no `e2e/` directory). There is no `.mcp.json` either, so an interactive agent cannot drive a browser.
- `commit-msg` is not in `.husky/`. Conventional Commits is mandated by CLAUDE.md but unenforced.
- Verify steps are scattered across `apps/api/Makefile`, `apps/web/package.json` scripts, and CI. There is no single command an agent can run before declaring work done.
- Biome's `apps/web/biome.json` disables `noUnusedVariables` and `noUnusedImports` for `*.vue` due to a known SFC limitation; nothing currently catches unused script-setup bindings.
- No production runtime feedback signal (`Sentry` not present in either tree).

This is a portfolio project, single developer, single user. Constraints favour cheap, well-known tools that integrate with existing husky + uv + biome + ruff + vitest + alembic infrastructure rather than introducing parallel pipelines.

## Goals / Non-Goals

**Goals:**

- Make every silent-drift class of bug surface as a failing local command or CI step:
  - FE consuming a stale BE response shape → TypeScript compile error.
  - Python type-hint regression → `basedpyright` failure.
  - Broken Vitest spec → CI failure.
  - Broken UI golden path → Playwright failure (locally + CI).
  - Off-convention commit message → rejected commit.
- Provide a single `verify` entry point that runs the entire backpressure suite, so an agent has one command to run before claiming done.
- Give an interactive agent (Claude Code) the ability to drive the browser via MCP Playwright, matching the workflow described in the "with backpressure" presentation slide.
- Keep local feedback fast: pre-commit stays staged-only; pre-push stays under ~30s; the heavier `verify` is opt-in.

**Non-Goals:**

- Replacing existing tools (no Ruff → mypy migration, no Biome → ESLint migration). We add the smallest set of *new* tools, configured to complement the existing ones.
- Test-coverage thresholds, mutation testing, Storybook/visual regression. Out of scope.
- Backend type-checker strictness on third-party stubs. We accept SQLModel/SQLAlchemy quirks via targeted ignores.
- Unifying the frontend HTTP layer beyond what `openapi-fetch` requires (no big composable rewrite, no Pinia restructuring).
- Sentry features beyond minimal init (no performance tracing, no session replay, no source-map upload — can come later).
- Pre-push running tests. Tests run only in CI and on-demand via `verify` to keep pre-push fast.

## Decisions

### 1. Typed HTTP client: `openapi-fetch` (not Orval, not Hey API, not hand-wrapped `$fetch`)

`openapi-fetch` is a tiny (~5kB) runtime that types `path` + `method` + body + response from the same `types/api.d.ts` we already generate via `openapi-typescript`. Same toolchain, same author, zero codegen churn beyond what we already do.

- **Alternative — Orval / Hey API / Kubb:** generate full client modules per path. More magic, larger output, would force a directory structure decision and more churn in `apps/web/types/`. Rejected: `openapi-fetch` is the minimal increment.
- **Alternative — keep `$fetch`, add stricter hand-written types via `paths['/x']['get']['responses']['200']` indexing:** verbose, easy to skip, doesn't fix the root cause (call sites still untyped on path/method).
- **Alternative — tRPC:** explicitly forbidden by the architecture rules (REST-only).

A thin wrapper (`apps/web/app/utils/api.ts`) exports a singleton `$api` configured with `runtimeConfig.public.apiUrl` and `credentials: "include"`. Used directly inside `$fetch`-style call sites; we do not invent new composables to wrap it. `useFetch` cases (which need SSR data fetching) wrap `$api.GET(...)` inside `useAsyncData` — a small, documented pattern.

### 2. Python type checker: `basedpyright` (not mypy, not pyright)

`basedpyright` is a community fork of pyright with saner defaults and active development; runs via `uv run basedpyright` so we stay inside the existing uv toolchain. Single binary, no separate venv.

- **Alternative — mypy:** widely used but slower on a SQLModel codebase, weaker SQLAlchemy 2.0 support, and the SQLModel author himself recommends pyright. Rejected.
- **Alternative — pyright (upstream):** fine, but `basedpyright` provides the same engine plus stricter defaults and faster release cadence. Either works; we pick one and move on.

Configured `strict` for `app/`, `standard` for `tests/` and `alembic/versions/`. Targeted ignores in `pyproject.toml` for SQLModel relationship-attribute typing quirks rather than relaxing the global level.

### 3. Playwright + MCP for UI verification

`@playwright/test` installed in `apps/web/`. One smoke spec (`apps/web/e2e/smoke.spec.ts`) covers: anonymous user lands on `/`, logs in via mock OAuth, opens a recipe detail page, asserts cooking-mode core elements render. Runs against `nuxi preview` in CI; against `nuxi dev` locally.

`.mcp.json` at repo root exposes the Playwright MCP server (`@playwright/mcp`) so Claude Code can launch a browser, navigate, and screenshot interactively while iterating on UI — matching the presentation's "Tool: Bash | tsc → Fix Errors → MCP: Playwright | Screenshot → Fix Styling" loop.

- **Alternative — Cypress:** heavier, separate ecosystem, no first-party MCP server.
- **Alternative — Vitest browser mode:** good for component tests, not for the cookies/SSR/OAuth-redirect path.
- **Alternative — only MCP, no committed e2e spec:** loses the CI signal. We want both.

OAuth in CI: the smoke spec uses a `?dev_login=<email>` shortcut already supported by the API in dev mode (or — if not — we add a test-only middleware bypass gated behind `ENV != "production"`). The exact mechanism is the first task's open question; we will pick the path of least change.

### 4. Single `verify` entry point at repo root

Adding a top-level `npm run verify` (and a parallel `make verify`) that runs:

```
apps/api: ruff check + ruff format --check + basedpyright + pytest + alembic check
apps/web: biome check + nuxi typecheck (with --noUnusedLocals) + codegen-drift + vitest + playwright
```

Implementation: a small shell script `scripts/verify.sh` invoked by both the npm script and the Makefile target. Steps fail fast (`set -e`); each step prints a banner so an agent can quickly see where it broke.

- **Alternative — Turbo / Nx:** overkill for two packages.
- **Alternative — `pre-commit` (the framework, separate from husky):** would replace the existing husky setup; rejected as it's pure churn.
- **Alternative — only document the commands in CLAUDE.md, no script:** loses the "single command" property the slides emphasise. Rejected.

### 5. Vue unused-var detection: `vue-tsc --noUnusedLocals`, no second linter

Add `"noUnusedLocals": true` and `"noUnusedImports": true` to the Nuxt-generated tsconfig via `nuxt.config.ts`'s `typescript.tsConfig` override. `nuxi typecheck` already runs `vue-tsc`; this catches unused script-setup bindings without standing up an ESLint pipeline.

- **Alternative — add `eslint-plugin-vue` for `*.vue` only:** introduces a second linter, second config, second CI step. Rejected as redundant given `vue-tsc` already parses SFCs.
- **Alternative — turn the Biome rules back on:** the underlying Biome limitation hasn't changed; would produce false positives.

### 6. Sentry: optional, env-gated, minimal

Sentry SDKs (`sentry-sdk[fastapi]`, `@sentry/nuxt`) installed and initialised only when `SENTRY_DSN` is set. No performance tracing, no replay, no source-map upload in this change. Init lives in `apps/api/app/main.py` and `apps/web/app/plugins/sentry.client.ts` (client-only — SSR error wiring deferred). Local dev is unaffected because no DSN is present.

- **Alternative — skip Sentry, add structured logging only:** the slides explicitly call out "runtime feedback" as a backpressure pillar; Sentry is the cheapest way to honour that and the portfolio narrative benefits.
- **Alternative — full Sentry rollout (tracing, replay, releases, source maps):** more value but more configuration; defer to a follow-up.

### 7. Commit-msg enforcement: `commitlint` + `@commitlint/config-conventional`

Standard combo, integrates with husky in two lines (`npx --no -- commitlint --edit "$1"` in `.husky/commit-msg`). Configured at repo root as `.commitlintrc.cjs`.

- **Alternative — `cz-cli`/Commitizen prompt:** interactive, breaks pasted commits and tooling like `git commit -F`. Rejected.

## Risks / Trade-offs

- **Migrating ~12 call sites to `openapi-fetch`** → likely a handful of TypeScript errors in unrelated places (we discover ones the hand-typed shape was hiding). *Mitigation:* PR is large but mechanical; each call site is independent, tested via existing `useFetch` paths and the new Playwright smoke spec.
- **`basedpyright` will surface a backlog of latent type errors on first run** → the initial PR may need targeted `# pyright: ignore[...]` lines or per-file-ignore configuration. *Mitigation:* land the tool in a permissive (`standard`) mode first across the whole tree, then tighten `app/` to `strict` once the noisy warnings are addressed; both happen inside this change but as separate tasks so we can ship if `strict` proves too noisy.
- **Playwright in CI doubles the web-job runtime** (browser download + install). *Mitigation:* cache `~/.cache/ms-playwright`; only run a single `chromium` smoke spec; no cross-browser matrix.
- **Mock OAuth for the Playwright smoke spec** could leak into prod if the bypass isn't tightly env-gated. *Mitigation:* the bypass is `ENV != "production"` AND requires a specific test-only header / query param; covered by an explicit test in `apps/api/tests/test_auth.py`.
- **Sentry in dev with a stray DSN** could exfiltrate dev errors. *Mitigation:* only initialise when `SENTRY_DSN` is set AND `SENTRY_ENVIRONMENT` is provided; document this in `.env.example` and `DEPLOYMENT.md`.
- **`verify` becomes slow as the repo grows** and an agent stops running it. *Mitigation:* the script prints per-step durations; if any single step crosses ~30s we revisit (e.g., parallelise with `concurrently` or move Playwright behind a `--full` flag). Out of scope to optimise pre-emptively.
- **`vue-tsc --noUnusedLocals` will flag legitimate intentional unused destructures** (e.g., `const { someProp: _ } = props`). *Mitigation:* the `_` prefix convention disables the warning; codify in CLAUDE.md UI guidelines if it comes up more than twice.
